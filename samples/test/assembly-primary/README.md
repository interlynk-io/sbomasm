# Assembly Merge with Primary Examples

This directory contains example SBOMs demonstrating the `--assemblyMerge --primary` feature, which allows merging SBOMs into an existing primary SBOM without creating a new synthetic root.

## Overview

The `assembly merge with primary` feature is useful when:

- **Python wheels shipping JavaScript bundles**: Nest the JS SBOM inside the Python package's primary
- **Container images with bundled applications**: Keep the image's primary as root, nest application SBOMs
- **Any scenario where one SBOM should contain others as sub-components**

## How It Works

1. Primary SBOM's primary component becomes the document root (no new synthetic primary)
2. Secondary SBOMs' primaries become sub-components (`metadata.component.components`)
3. Secondary primaries are also added as direct dependencies of the primary component
4. Primary SBOM's identity (serial number, supplier, licenses, authors) is preserved
5. Timestamp is updated to reflect modification time

## Example SBOMs

### `primary.cdx.json`

A Java/Maven application SBOM with:

- **Primary Component**: `primary-app` (application)
- **Components**: `lib1`, `lib2` (Maven libraries)
- **Dependencies**: primary-app → [lib1, lib2]
- **Serial Number**: `urn:uuid:1111...` (preserved in output)
- **Supplier**: "Primary Supplier"
- **License**: MIT
- **Tools**: original-tool

### `secondary1.cdx.json`

An npm/JavaScript package SBOM with:

- **Primary Component**: `secondary-pkg` (library)
- **Components**: `dep-a`, `dep-b` (npm dependencies)
- **Dependencies**: secondary-pkg → [dep-a, dep-b]

### `secondary2.cdx.json`

A Python package SBOM with:

- **Primary Component**: `tertiary-lib` (library)
- **Components**: `dep-c` (PyPI dependency)
- **Dependencies**: tertiary-lib → [dep-c]

## Example Commands

### Example 1: Single Secondary SBOM

Merge one secondary SBOM into the primary:

```bash
sbomasm assemble --assemblyMerge \
  --primary primary.cdx.json \
  secondary1.cdx.json \
  -o output-single.cdx.json
```

**Expected Output Structure:**

```json
{
  "serialNumber": "urn:uuid:11111111-1111-1111-1111-111111111111",
  "metadata": {
    "component": {
      "name": "primary-app",
      "components": [
        { "name": "secondary-pkg" }
      ]
    }
  },
  "components": [
    { "name": "lib1" },
    { "name": "lib2" },
    { "name": "dep-a" },
    { "name": "dep-b" }
  ],
  "dependencies": [
    {
      "ref": "primary-app",
      "dependsOn": ["lib1", "lib2", "secondary-pkg"]
    }
  ]
}
```

### Example 2: Multiple Secondary SBOMs

Merge multiple secondary SBOMs into the primary:

```bash
sbomasm assemble --assemblyMerge \
  --primary primary.cdx.json \
  secondary1.cdx.json \
  secondary2.cdx.json \
  -o output-multiple.cdx.json
```

**Key differences from regular assembly merge:**

- Primary SBOM's serial number is preserved (not a new UUID)
- Primary's supplier, licenses, and authors are maintained
- Timestamp is updated to current time
- sbomasm tool is added to the tools list

### Example 3: Comparison with Regular Assembly Merge

**Without `--primary` (creates new synthetic root):**

```bash
sbomasm assemble --assemblyMerge \
  -n "new-app" -v "1.0" -t "application" \
  primary.cdx.json secondary1.cdx.json \
  -o regular-assembly.cdx.json
```

**With `--primary` (uses existing primary as root):**

```bash
sbomasm assemble --assemblyMerge \
  --primary primary.cdx.json \
  secondary1.cdx.json \
  -o assembly-with-primary.cdx.json
```

| Aspect | Without `--primary` | With `--primary` |
|--------|-------------------|------------------|
| Primary | New synthetic "new-app" | "primary-app" (from primary SBOM) |
| Serial Number | New UUID | Preserved from primary |
| Supplier | Not set (or from flags) | "Primary Supplier" (from primary) |
| License | Not set (or from flags) | MIT (from primary) |
| Sub-components | Both input primaries | Only secondary primaries |

## Real-World Use Case: Python Wheel with JavaScript

Imagine you have:

- `python-wheel.cdx.json` - Python package SBOM
- `js-bundle.cdx.json` - JavaScript bundle SBOM (generated by webpack)

```bash
sbomasm assemble --assemblyMerge \
  --primary python-wheel.cdx.json \
  js-bundle.cdx.json \
  -o complete-wheel.cdx.json
```

**Result:**

- Python wheel's primary remains the document root
- JavaScript bundle becomes a sub-component (assembly) of the Python wheel
- JavaScript bundle is also listed as a dependency of the Python wheel
- Perfect for compliance requirements showing bundled dependencies

## Verification

Check that the output preserves the primary's identity:

```bash
# Verify serial number is preserved
cat output.cdx.json | jq '.serialNumber'

# Verify primary component is from primary SBOM
cat output.cdx.json | jq '.metadata.component.name'

# Verify secondary is nested as sub-component
cat output.cdx.json | jq '.metadata.component.components[].name'

# Verify secondary is a dependency
cat output.cdx.json | jq '.dependencies[0].dependsOn'

# Verify original tools are preserved + sbomasm added
cat output.cdx.json | jq '.metadata.tools.components[].name'
```

## Notes

- Component deduplication still applies (duplicate components only appear once)
- BOM-refs are resolved and updated if conflicts occur
- All dependency relationships are preserved and updated
- The `--primary` flag is only valid with `--assemblyMerge`
- Primary SBOM must have a primary component in `metadata.component`
- Input arguments are secondary SBOMs only (primary is specified via `--primary`)
